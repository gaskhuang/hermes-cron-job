# ZoneTech Daily LINE Report

產生 ZoneTech 每日 LINE / LINE@ 報告並發布到 GitHub Pages。

## 預設值
- 目標日期：Asia/Taipei 前一天 00:00:00–23:59:59
- LINE@ bot id: 【你的LINE-BOT-ID】
- GitHub repo: 【你的GitHubUser】/【你的Repo】
- Pages base: https://【你的GitHubUser】.github.io/【你的Repo】/
- 排程：每天 10:00 和 17:00

## 工作流程

### 1a. Cron 快取路徑：檢查報告是否已存在
```bash
TZ=Asia/Taipei HHMM=$(date '+%H%M')
TZ=Asia/Taipei YMD=$(date '+%Y-%m-%d')
HTTP_CODE=$(curl -s -o /dev/null -w '%{http_code}' \
  "https://【你的GitHubUser】.github.io/【你的Repo】/web/line_official_${YMD}_${HHMM}.html")
```
- 200 → 驗證內容日期正確，跳過步驟 3-7
- 404 → 正常執行

### 1. 計算日期
```bash
TZ=Asia/Taipei date '+%Y-%m-%d %H:%M:%S %Z %z'
```

### 2. 預檢
```bash
command -v opencli >/dev/null 2>&1 && opencli doctor
```
若 opencli doctor 顯示 daemon + extension connected → 使用 line-opencli-all skill 提取。不需要 CDP。

### 3. LINE@ 提取
- 確認 【LINE後台網址】 頁面已登入
- 使用 in-page fetch() 帶 browser cookies
- 聊天列表：/api/v2/bots/{BOT}/chats?folderType=ALL&limit=25，用 next 分頁
  - 回傳陣列在 .list，不是 .chats
  - 顯示名稱在 c.profile.name
- 候選聊天室訊息：/api/v3/bots/{BOT}/chats/{chatId}/messages?limit=25
  - 事件陣列在 .list，不是 .messages；事件文字在 m.message.text
  - 分頁 cursor 在 resp.backward（response 欄位，不是 request param）
  - m.timestamp = 事件時間 ms；m.type === "message" = 客戶；"messageSent" = 我方
- 只包含目標日期內有實際事件的聊天室

### 3a. 預算檢查
```bash
CANDIDATE_COUNT=$(python3 -c "import json; d=json.load(open('/tmp/report_gen/line_chat_list.json')); print(len(d))")
```
- ≤20 → 完整提取；21-50 → batch 單頁；51-80 → 僅優先；>80 → metadata-only

### 5. 去敏化
- 遮罩：手機、市話、email、密碼、銀行帳號、地址（僅保留縣市）、發票 URL、內部 IP、設備識別碼
- 掃描最終 MD+HTML 確認無敏感資訊
- 絕不提交原始 API 回應或中間 JSON

### 6. 產生報告
報告結構：
1. # LINE@ 官方帳號客戶訊息報告 — YYYY-MM-DD HH:MM
2. 目標訊息範圍：昨日 00:00–目前
3. 資料來源狀態 + GitHub Pages URL
4. 緊急需回覆 / 高風險（最多 10 項）
5. 依案件狀態整理（已完成, 待收款, 工程需處理, 客服/業務需回覆, 需人工確認）
6. 依分類整理所有客戶訊息
7. 明確待辦清單（僅公司方動作）
8. 可信度與限制

### 分類優先順序（V3）
1. 客戶最後文字為結尾語 → 已完成/不需處理（可以了, 沒問題, 收到, 好的, 謝謝, 了解, ok, 完成了）
2. 超過 7 天無文字 → 已完成/不需處理（除非有新請求）
3. Done label → 已完成/不需處理
4. 我方最後發言 → 已完成/不需處理（除非含主動承諾：我會再, 稍後回, 工程會, 我來安排）
5. 客戶最後發言 → 依文字分類：
   a. 緊急（斷線/當機/連不上/中毒）→ 工程需處理 / 緊急技術/客訴
   b. 財務（付款/匯款/發票/帳款/尾款/轉帳/查收）→ 待收款 / 催款
   c. 工程（安裝/維修/設定/施工/路由器/switch/ap/nas/防火牆/wifi）→ 工程需處理
   d. 業務（報價/詢價/預約/規劃/採購）→ 客服/業務需回覆 / 報價/商機
   e. 有意義但無類別（>5 字）→ 一般客服
   f. 極短/僅附件 → 需人工確認

### 7. 發布
- 產生到 /tmp/report_gen/ → clone repo 到 /tmp/zonetech-daily-report/
- 複製 MD → repo/reports/，HTML → repo/web/
- 更新 index.html 和 README.md
- commit + push
- 輪詢 Pages 直到 HTTP 200

## 關鍵坑點

- macOS 安全限制：禁止 pipe-to-interpreter 模式（curl | python3）。改用 file intermediary：curl > file，再 python3 file
- 去敏 sanitize() 必須在每個訊息文字路徑呼叫，包括 summary 欄位和 HTML 版
- 密碼常以「密碼XXXX」無冒號格式出現，需用 r'密碼\s*[A-Za-z0-9@#$%!?]{6,}' 匹配
- 訊息排序方向：API 回傳 newest-first，分類前需 sort((a,b)=>a.timestamp-b.timestamp)
- toLocaleString 日期格式為 2026/6/12（斜線無前綴零），與 strftime('%Y-%m-%d') 不同，日期比較一律用 epoch ms

# LINE OpenCLI 全路徑（唯一方案）

## 觸發關鍵字
LINE@ 登入不了、LINE 無法登入、登入失敗、session 過期、OAuth 登入 → 檢查 extension 狀態，依 Recovery 流程重新登入。

## 適用場景
- LINE@/LINEBiz 官方帳號（主力）：聊天記錄查詢、聊天清單抓取、客戶對話整理、未回覆查詢、回覆客戶訊息
- 個人 LINE（Chrome Extension 備援）：傳訊息給特定聯絡人
- LINEBiz Chrome Profile 管理：backup/restore、watchdog、CDP port 切換、auto-login recovery

## 核心原則
首選 OpenCLI，CDP 專用 Profile（port 9228）為備援。
正常情況走 OpenCLI（extension connected）。若 OpenCLI extension 斷線，
用 Dedicated Chrome Profile（port 9228, ~/ChromeProfiles/LineBiz/）透過 CDP WebSocket 操作。

## 前置條件
- macOS 本機
- OpenCLI v1.8.0+ 已安裝、daemon running、extension connected
- Chrome 已登入 LINE Business ID
- Session name 統一用 linebiz

## Recovery: Extension 斷線／Chrome 重啟復原

### 0a. 確認 Chrome 行程狀態
```bash
ps aux | grep ChromeProfiles/LineBiz | grep -v grep
```
- 行程不存在 → 執行 0b 重啟 Chrome
- 行程存在但 extension 未連 → 執行 0c 確認 daemon 狀態
- 行程存在且 port 9228 可連 → 跳至 0d（CDP 確認分頁）

### 0b. 啟動 Chrome（含 extension + debug port）
```bash
kill $(pgrep -f "ChromeProfiles/LineBiz") 2>/dev/null; sleep 1

/Applications/Google\ Chrome.app/Contents/MacOS/Google\ Chrome \
  --remote-debugging-port=9228 \
  --remote-allow-origins=* \
  --user-data-dir=/Users/【你的帳號】/ChromeProfiles/LineBiz \
  --profile-directory=Default \
  --no-first-run \
  --disable-features=ChromeWhatsNewUI \
  --load-extension=/Users/【你的帳號】/Downloads/opencli-extension-v1.0.9 \
  > /tmp/chrome_linebiz_log/stdout.log 2>&1 &
sleep 4
```
⚠️ --remote-allow-origins=* 是 Chrome 148+ 必要參數。不加則 WebSocket 連線會收到 403 Forbidden。

驗證：
```bash
ps aux | grep ChromeProfiles/LineBiz | grep -v grep | wc -l
# 確認 ~10 個行程表示 Chrome 正常啟動

curl -s http://localhost:9228/json/version | python3 -c "import sys,json; d=json.load(sys.stdin); print(d.get('Browser','?'))"
```

⏳ Extension 初始化時間：Chrome 啟動後約需 3–5 分鐘才會完成背景初始化並連上 daemon。不要急著重啟。

### 0c. 重啟 OpenCLI daemon
```bash
opencli daemon restart
sleep 3
opencli doctor
```
若仍失敗：
```bash
pkill -f "opencli.*daemon" 2>/dev/null
sleep 2
opencli daemon start
sleep 3
opencli doctor
```

### 0d. CDP 直接確認分頁
```bash
curl -s http://localhost:9228/json | python3 -c "
import json, sys
pages = json.load(sys.stdin)
for i, p in enumerate(pages):
    print(f'{i}: {p.get(\"title\",\"?\")[:50]} | {p.get(\"url\",\"?\")[:80]}')
"
```
若無 LINE 相關分頁，開新分頁：
```bash
curl -s -X PUT "http://localhost:9228/json/new?https://manager.line.biz/account/【你的LINE帳號ID】"
```

### 0e. 檢查是否需 OAuth 登入
新開分頁若跳轉到 access.line.me 或 account.line.biz/login，表示 session 失效。執行 OAuth 2 次點擊登入流程。

### 0f. 區分「Extension 斷線」vs「OAuth token 過期」
- Extension 斷線：opencli doctor 顯示 Extension: not connected → 走 0a–0c Chrome/daemon 重啟流程
- OAuth token 過期：opencli doctor 顯示正常，但 watchdog 顯示 chat_api_not_authenticated → 不需重啟 Chrome，直接走 0e → 4a OAuth 登入

### 6. API 提取聊天資料（進階）

#### 6a. 取得 bot ID
從 URL 取得：https://【LINE後台網址】/【你的LINE-BOT-ID】
→ bot ID = 【你的LINE-BOT-ID】

#### 6b. 取得聊天列表（API）

- 分頁用 &next=cursor（cursor 原始字串，非 URL）
- 回傳欄位：chatId, profile.nickname, updatedAt, lastReceivedAt, lastSentAt, read, done, latestEvent

**緊湊摘要模式：**


#### 6d. 取得單一聊天室訊息
```bash
opencli browser linebiz eval "fetch('/api/v3/bots/【你的LINE-BOT-ID】/chats/{CHAT_ID}/messages?limit=25').then(r=>r.json()).then(j=>JSON.stringify(j).slice(0,8000))"
```
- 初始請求不用加 backward 參數
- 往前翻頁用 ?backward=cursor
- 事件類型：message（客戶發送）、messageSent（我方發送）
- source.userId === BOT_ID → 我方發出
- timestamp 為 epoch ms，需轉換 Asia/Taipei (+08:00)
- .list 欄位為事件陣列，不是 .messages
- 事件文字在 m.message.text，不是 m.text

#### 6e. 全量提取最佳實踐
1. 用 write_file 寫入完整 JS 腳本
2. base64 編碼後注入 — 繞過 shell substitution 問題
3. 批次並行 fetch — Promise.all(batchSize=5)，約 20–30 秒完成 80 個 chat
4. 結果存 window.__xxx — 避免 eval 輸出截斷
5. 兩段式取回 — 先驗證變數存在，再 eval "window.__xxx" > file.json

### 8. CDP WebSocket 直接 API 存取（OpenCLI extension 斷線時備援）

當 opencli doctor 顯示 Extension: not connected，但 Dedicated Chrome Profile（port 9228）仍在運行時，可直接透過 CDP WebSocket 執行 JavaScript 呼叫 LINEBiz 內部 API。

前置條件：Chrome 以 Dedicated Profile 啟動並綁定 debugging port：
~/ChromeProfiles/LineBiz/ + --remote-debugging-port=9228 + --remote-allow-origins=*

#### 8a. 檢查 CDP 狀態
```bash
curl -s http://localhost:9228/json/version | python3 -c "import sys,json; d=json.load(sys.stdin); print('Chrome:', d.get('Browser','?'))"
```

#### 8b. 透過 WebSocket 執行 JS 取得 API 資料
需安裝 websocket-client：pip3 install websocket-client

範例腳本 — 取得 LINE@ 聊天列表摘要：
```python
import json, urllib.request, websocket

resp = urllib.request.urlopen('http://localhost:9228/json')
pages = json.loads(resp.read())
line_page = [p for p in pages if '【LINE後台網址】/【你的LINE-BOT-ID前綴】' in p.get('url','')]
page = line_page[0]
ws = websocket.create_connection(page['webSocketDebuggerUrl'], timeout=10)

cmd = json.dumps({
    'id': 1,
    'method': 'Runtime.evaluate',
    'params': {
        'expression': 'fetch("https://【LINE後台網址】/api/v2/bots/【你的LINE-BOT-ID】/chats?folderType=ALL&limit=25&prioritizePinnedChat=true").then(r=>r.json()).then(j=>({total:j.list.length,unread:j.list.filter(c=>!c.read).length,undone:j.list.filter(c=>!c.done).length,unfollowed:j.list.filter(c=>!c.followedUp).length})).then(d=>JSON.stringify(d))',
        'returnByValue': True,
        'awaitPromise': True
    }
})
ws.send(cmd)
resp = json.loads(ws.recv())
ws.close()

if 'result' in resp and 'result' in resp['result']:
    print(resp['result']['result']['value'])
```

#### 8d. macOS 安全限制：pipe-to-interpreter 被攔截
macOS（Sonoma+）的安全掃描會攔截 curl url | python3 -c "..." 的 pipe-to-interpreter 模式。
強制解法：一律使用 file intermediary，不要用 pipe：
```bash
# ❌ 會失敗
curl -s http://localhost:9228/json | python3 -c "..."

# ✅ 分兩步，安全
curl -s http://localhost:9228/json > /tmp/cdp_pages.json
python3 -c "import json; ..."  # 從檔案讀取
```

#### 8e. 注意事項
- Runtime.evaluate 的 params 需含 "awaitPromise": true 才能正確等待 fetch 回傳
- expression 中的雙引號在 Python 字串中需跳脫為 \\"
- 每次建立新 WebSocket 連線，不需擔心變數名稱衝突
- WebSocket 連線約 80 次 Runtime.evaluate 後會斷線。大量提取（50+ chats）時，建議每 15 個 chat 重新連線一次
- 若分頁導航（window.location 變了），WebSocket URL 可能失效，需重新取得

## 注意事項
- OpenCLI v1.8.0+ 必須加 session name：opencli browser linebiz open URL
- 沒有 opencli browser navigate — 用 opencli browser open
- 沒有 opencli browser snapshot — 用 opencli browser state
- opencli browser click 不支援文字匹配 — 用 opencli browser eval 執行 document.querySelector(...).click()
- JS 注入時若用 $(cat) 或直接 shell 字串要避免 // 註解 — shell substitution 會吃掉它造成 SyntaxError。用 /* */、minified，或改用 base64 編碼（推薦，// 安全）
- 若 opencli browser eval 回傳空/exit 1，不代表執行失敗 — 檢查 window.__lineReport 是否存在
- 複雜 JS 注入優先使用 base64 編碼：SCRIPT_B64=$(base64 < script.js); opencli browser linebiz eval "$(echo "$SCRIPT_B64" | base64 -d)"
- 變數名稱跨 eval 持續存在：同頁面 context 下第二次宣告 const bot=... 會出 SyntaxError。使用唯一名稱或重新載入頁面
- base64 注入時 fetch() 需用絕對 URL：inline 單行可用相對路徑，但 base64 解碼後的多行 async 腳本中相對 URL 會拋出 Failed to parse URL。一律用 const BASE='https://【LINE後台網址】'; fetch(BASE+'/api/...')
- 從 window.__lineReport 擷取大 JSON：opencli browser linebiz eval "window.__lineReport" > /tmp/file.json
- 不讀、不傳、不外洩 LINE 對話中的敏感客戶資料

## 重要坑點
- Google News sidecar 嚴格 24h  only — 每日 Hermes vs OpenClaw 新聞摘要只包含過去 24 小時的新聞
- 平行搜尋執行取決於 tool host — 若 terminal 拒絕 shell & 背景化，使用 terminal(background=true) 呼叫並輪詢輸出
- opencli JSON 輸出包含 trailing update messages — 剝離 "Update available..." 等 npm 通知後再解析
- opencli JSON 形狀變化：raw list 或 {"items": []}；xreach 可能回傳 [[...]] 或 [...]
- 不要用 cat file.json | python3 -c "..."（pipe-to-interpreter 被 Hermes 安全掃描封鎖）
- 使用 execute_code 搭配 open() + json.load() 或 write_file 寫入腳本後 terminal 執行
- 報告輸出為 .txt 供 NotebookLM 使用 — NotebookLM 處理 .txt 比 .md 更可靠
- Cron 模式 execute_code 被封鎖 — 改用 write_file 寫入腳本，再 terminal("python3 /tmp/script.py") 執行
- opencli created_at vs xreach createdAt — 使用正確的欄位名稱和解析格式
- opencli views 欄位為字串 '1' 非 int 1 — 使用 safe_int() helper
- q3 shell & + wait 成功率僅 67%；Python subprocess.run() serial 成功率 100% — 優先使用 Python subprocess
- Stale q3 檔案可能從先前執行殘留 — 檢查 mtime 或強制重新生成
- 報告標頭日期欄位必須匹配實際計算的 windows
- 合併 JSON 輸出結構可能為 dict {unique: [...]} 而非 flat list — 讀取時用 data['unique']
- 細節檔案以數字 tweet ID 命名 — 用 os.listdir() 迭代而非猜測檔名

## Sampling rule for digests
Target ~200 timeline items minimum before filtering.
Keep paginating until you reach ~200 items or hit practical limits.

## Filtering guidance
After fetching the large sample, filter out:
- ads / promotions / coupons / giveaway posts
- vague hype posts with no concrete claim
- duplicate reposts of the same idea
- low-information quote tweets like "so true"
- old evergreen/promoted-looking posts
- engagement-bait posts ("Like + Repost + Comment", "Follow me so I can DM")

Prioritize:
- AI tooling / model / infra updates
- semiconductor / market / security signals
- high-engagement posts with concrete product, pricing, policy, or workflow changes
- recurring themes appearing across multiple posts

**Theme classification sanity rule:** Do not let broad AI/agent terms make a post appear under unrelated buckets. For semiconductor/hardware, require hardware-specific terms (NVIDIA, GPU, TSMC, HBM, CPU, ASIC, etc.). For security, require security-specific terms (CVE, breach, token leak, exploit, malware, etc.).

## Practical scoring heuristic
- start from likeCount + 2*retweetCount + 1.5*quoteCount + 0.2*replyCount + 1.2*bookmarkCount + 0.0005*viewCount
- down-rank replies heavily
- down-rank retweets and quote-only posts unless quoted content is central
- down-rank posts older than desired recency window
- down-rank bare URL / image link-only posts
- keep medium-engagement posts if unusually concrete and relevant
- engagement alone is NOT enough: require topical relevance first
- explicitly filter off-topic high-engagement items (religion, memes, entertainment, sports)

## Recency / feed-quality pitfalls
Add two guards before summarizing:
1. Recency window: usually keep roughly last 48-72 hours
2. Topic gate: require clear relevance to digest domains before using engagement for ranking

- Use artifact list as PRIMARY status check, but treat download as final readiness probe

- Retry-while-pending unsticks stalled tasks: launch second generate call without cancelling first
- Cron job context_from injection floods context window — mitigation: cron prompt must explicitly instruct agent to IGNORE context_from content and only read from file path
- Report file date ≠ cron execution date — upstream x-radar-report covers previous day's activity
- Background terminal processes silently drop stdout — use foreground terminal calls instead
- execute_code has 300s timeout — insufficient for tasks needing >5 min of polling
- Infographic generation follows same pattern as slide-deck/audio

  - Foreground fix: export PATH="$HOME/.local/bin:$PATH"
  - Background/cron fix: Remove the stale binary: rm "$(which notebooklm)"; then which notebooklm verifies it points to pipx ~/.local/bin/notebooklm
- pip show and notebooklm --version can disagree; inspect both
- The package is unofficial and uses undocumented Google APIs
- Rate limits and quota issues are normal for heavier generation tasks

## Classification Rules (v2 — Corrected 2026/5/28)

### ⚠️ Critical Step 1: Verify Activation Was Sent Before Classifying
**This is the #1 source of classification errors.** LINE@ chats are shared between sales AND support. A customer who only used the chat for technical support and said 「謝謝」 is NOT a sales lead.

Scan the last 25 events for activation patterns:
- 「好久不見！」 / 「老客戶開發」
- 「感謝XXX長期以來對蓋斯克服務的信任」
- 「距離上次合作已超過」

If NONE contain activation text: **this chat has no activation response → classify as 未回覆 regardless of customer replies.**

### Rule: Support-Only Chat → 未回覆
If the last events show pure support/admin with NO activation sent:
- Technical issue reports → resolution → customer says 「謝謝」 / 「好的」 / 「了解」
- Invoice / billing questions
- General admin (password reset, account info)
- Engineer visit scheduling
→ 未回覆 — no activation was delivered here.

### 未回覆 — Three Scenarios
1. **No customer reply after activation** — last event is from us.
2. **No activation detected** — chat is support-only, regardless of customer activity.
3. **Brush-off reply** — customer responded to activation but with a polite decline:
   - 「目前穩定」 / 「目前還行」 / 「目前OK」 / 「目前都穩定」 / 「目前都還算ok」
   - 「目前暫時應該沒有」 / 「目前沒」 / 「目前正常」 / 「目前解決了」
   - 「都還穩定」 / 「都還正常」 / 「沒有什麼問題」 / 「沒什麼問題」
   - 「使用上應該都還算順利」
   - 「沒有收到工地的回饋」 / 「沒有反應」
   - 「還行」 / 「還好」 / 「沒」
   - 「謝謝」 / 「感謝關心」 (as the only response to activation)
   **These are NOT 「有興趣」** — they are polite brush-offs. Classify as 未回覆.

### 有興趣 — Genuine Soft Positive
Use when customer replied and showed actual interest, but hasn't started concrete discussion:
- 「感謝資訊提供，我們評估一下」
- 「可以先給我資料」
- 「之後可以聊聊」 / 「可以聊聊」

**Do NOT classify as 有興趣 when:**
- Customer said 「目前穩定」 or equivalent brush-off
- The reply is purely acknowledging the message without indicating interest
- The conversation is entirely technical support

### 討論中 — Active Purchase Discussion
Use only when customer is actively discussing a concrete purchase:
- Asking for price: 「多少錢」 / 「報價」 / 「價格」 / 「費用」 / 「這個價格有變便宜了嗎」
- Asking for quotation: 「可以幫我估價」 / 「請報價」 / 「報價單」
- Requesting service: 「可以幫我處理」 / 「想了解...方案」
- Scheduling for purchase: 「約時間看一下」 / 「什麼時候可以」 / 「能幫我們評估」
- On-site installation discussions: 「工程師能到現場嗎」 / 「到府安裝」

### Key Anti-Patterns (What NOT to Do)
1. Don't confuse tech support with interest.
2. Don't confuse acknowledgement with enthusiasm.
3. Don't elevate engineering questions to sales.
4. Don't treat 「目前穩定」 as positive.
5. Don't classify 外包廠商. If a chat is known as 外包合作廠商, skip it entirely.

## Intent Scoring (1-10) for Phone Followup Prioritization

### Score Scale
| Score | Meaning | Action |
|-------|---------|--------|
| 9-10 | Actively asking about products/pricing | Call immediately |

### Scoring Algorithm
```python
customer_msgs = [m for m in last10 if not m['isSent']]
customer_texts = [m['text'] for m in customer_msgs if m['text'] not in ('[其他]', '[貼圖]')]

    '多少錢','報價','價格','費用','可以幫','想了解','需要','租','買','採購','能幫'])

    '約時間','這幾天','今天','明天','什麼時候'])

high_count = sum(1 for kw in purchase_high if kw in all_text)
medium_count = sum(1 for kw in purchase_medium if kw in all_text)

elif asking_products:              score = 9
elif high_count >= 3:              score = 8
elif high_count >= 2:              score = 7

elif high_count >= 1:              score = 6

elif medium_count >= 1:            score = 5
```

- The JavaScript code is too complex for the extension to handle synchronously
- The OpenCLI daemon itself times out the pipe

                        btn.classList.remove('copied');
                    }, 1500);
                }
                if (navigator.clipboard && navigator.clipboard.writeText) {
                    navigator.clipboard.writeText(text).then(done).catch(fallback);
                } else {
                    fallback();
                }
                function fallback() {
                    var ta = document.createElement('textarea');
                    ta.value = text;
                    ta.style.position = 'fixed';
                    ta.style.opacity = '0';
                    document.body.appendChild(ta);
                    ta.select();
                    try { document.execCommand('copy'); done(); } catch (e) {}
                    document.body.removeChild(ta);
                }
            });
            h2.appendChild(btn);
        });
